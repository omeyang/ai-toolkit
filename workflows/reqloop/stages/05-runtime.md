# 阶段 5 — 运行时校验 (runtime)

## 目标

在**指定环境**下实际编译、测试、lint，并产出"业务边界覆盖矩阵"。
**AI 不对"是否符合业务"下结论**，只如实报告。

## 输入

- 目标环境（用户指定：dev / staging / 具体 CI 流水线）
- 阶段 2 的代码 / MR

## 步骤

1. **触发 CI 流水线**
   - 调用 `qianliu-ci`，在**用户指定**的环境触发：
     - build
     - unit test (含 race / cover)
     - lint (golangci-lint / 对应语言的 linter)
     - （可选）漏洞扫描
   - 严禁自行切换到"更容易跑过"的环境

2. **轮询结果**
   - 失败日志必须拉回本地归档，不只是链接
   - 失败 → 在报告中标红，但**不中止**流程（让阶段 7 汇总）

3. **生成业务边界覆盖矩阵**

   这一步是本阶段的**核心产物**。对反讲文档中每条已 confirm 的业务场景，填：

   | 业务场景 | 代码路径 | 单测覆盖 | 集成测试覆盖 | 环境/数据覆盖 |
   |---------|---------|---------|------------|-------------|
   | 场景 A | ✅ repo-a/foo.go | ✅ foo_test.go | ❌ | ✅ dev |
   | 场景 B | ✅ repo-b/bar.go | ❌ | ❌ | ❓ |

   - ✅ 有证据（文件 / 测试名 / 环境名）
   - ❌ 确定缺失
   - ❓ 无法判定（需人工补充）

4. **AI 不下"符合业务"的结论**
   - 只报告矩阵
   - 任一行有 ❌ 或 ❓ → 在报告中标注"未闭环"
   - 是否放行由人工在阶段 7 决定

## 产物

`.reqloop/runtime-{id}.md`：

```markdown
---
id: {id}
stage_status: complete | in_progress
env: {用户指定环境}
ci_pipeline: {流水线链接}
---

## CI 结果
- build: ✅ / ❌
- test: ✅ / ❌ （覆盖率 X%）
- lint: ✅ / ❌ （N 条问题）

## 失败日志摘要
...

## 业务边界覆盖矩阵
[表格]

## 未闭环项
- 场景 B: 缺单测、缺环境验证
```

## 失败处理

- CI 触发失败（权限 / 配置） → 中止并提示用户
- 流水线超时 → 记录部分结果，标注 `partial: true`
